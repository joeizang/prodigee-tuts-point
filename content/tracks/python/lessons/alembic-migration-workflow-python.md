# Alembic Migration Workflow

## Learning objectives

- Explain the role of Alembic revisions.
- Distinguish model metadata from migration history.
- Plan upgrade and downgrade operations.
- Add review notes for data risk.
- Test migration expectations before production.

## Prerequisites

You should understand SQLite migrations, schema versions, and production startup checks.

## Mental model

**Term: Alembic revision** means a versioned migration file with an identifier, parent revision, upgrade, and downgrade.

**Term: autogenerate** means tooling compares model metadata with the database. It helps, but humans must review the result.

**Term: downgrade** means a planned reversal path. Even when production downgrade is restricted, the decision should be deliberate.

## Core idea

An Alembic migration is reviewed source code:

```python
def upgrade() -> None:
    op.add_column("notes", sa.Column("archived", sa.Boolean(), nullable=False, server_default="0"))
```

Do not blindly trust generated migrations.

## Worked example

Every revision should answer:

- What schema changes?
- Is existing data preserved?
- Can it be rolled back?
- What test proves the migration?

## Production transfer

Alembic becomes the real migration workflow when SQLAlchemy/SQLModel enters the app. The discipline learned with raw SQLite schema versions carries forward: review, test, apply in order, verify.

## Common mistakes

- Treating autogenerated migrations as automatically safe.
- Forgetting downgrade or documenting why it is impossible.
- Mixing data backfills with schema changes casually.
- Running migrations at request time.
- Skipping migration tests.

## Testing strategy

Test the generated revision plan as a contract first. Later, run migration files against temporary databases.

## Debugging strategy

If migration state is confusing, inspect revision id, down revision, upgrade operations, downgrade operations, and the database's current revision.
