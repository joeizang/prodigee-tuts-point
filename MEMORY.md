# Project Memory

## Product Quality

- The learning content must be book-depth, pedagogically strong, and senior-engineer useful, not shallow tutorial filler. Treat the referenced books as a quality benchmark for scope, rigor, sequencing, conceptual explanation, examples, exercises, and projects, while producing original curriculum text and code. Content must teach through clear mental models, worked examples, misconception checks, progressive practice, reflection prompts, and project transfer.
- All code examples in learning content must be syntax highlighted by default. Plain unhighlighted code blocks are a content-quality regression unless the block is intentionally raw terminal/output text.
- Important terms in learning content should be visually emphasized and explainable in context. Use bold/highlighted key terms plus compact term bubbles or popovers for deeper definitions, examples, misconceptions, and project relevance.
- Do not leave frontend route shells as monolithic files once a slice grows. Keep App.tsx focused on app shell/routing, with pages, shared components, feature components, and hooks split into maintainable modules.
- Primary action buttons must have explicit readable contrast in every theme, including icon and text color. Do not rely on inherited button colors when the action background changes.
- Sidebar navigation must remain compact and scan-friendly. Do not let grid or flex children stretch nav rows vertically; explicitly pin nav lists to start alignment with fixed/min-content row sizing.
- Embedded editors must follow the selected app theme immediately when the user switches themes, not only after remount/reload. Light, neutral, and dark modes need distinct editor themes; neutral must use a Nord Light-style Monaco palette rather than Monaco's default light/dark themes or a washed-out generic gray theme.
- Do not use OmniSharp as the C# IntelliSense backend for this project. Prefer direct Roslyn-backed services for C# exercise workspaces, and document any future full implementation as Roslyn/LSP-style rather than OmniSharp-bound. The C# editor standard is the Microsoft C# VS Code extension experience except code navigation is not required yet: semantic IntelliSense, current-buffer accurate locals/parameters, diagnostics/linting/squiggles, formatting, and refactoring/code actions. Apply the same quality bar to Swift and TypeScript using their proper language services. Do not ship fake suggestions, stale editor state, noisy member-access completions, or generic Monaco item icons when semantic tags are available.
- Treat editor language-service latency as part of correctness. Completion must be designed for keystroke-speed use with measured cold and warm timings; avoid per-request design-time project loads, pre-warm or cache expensive Roslyn/language-service state where practical, and do not call a language-service slice complete if the popup feels sluggish in normal typing.
- Do not mark editor-language acceptance criteria complete when the implementation is a placeholder, static message, client-side fake, or partial heuristic. A checked language-service criterion must be backed by the proper semantic service and at least one regression test or explicit verification.
- When changing generated exercise workspace contracts, migrate or invalidate existing persisted workspaces. Passing tests with a fresh profile is not enough; stale `App_Data/exercise-workspaces/<profile>/<exercise>` folders can keep serving wrong file names, wrong language-service messages, and wrong editor modes after a restart.
- Global curriculum surfaces must be active-track scoped unless explicitly designed as cross-track. Sources, review cards, navigation, dashboard shortcuts, and similar aggregate views should pass `trackId`, render only that track's objects, and have regression coverage for non-C# tracks such as Python or Swift.
- For SourceKit-LSP and any future stdio LSP bridge, implement JSON-RPC framing byte-accurately because `Content-Length` counts UTF-8 bytes, not characters. For Swift diagnostics, do not rely only on `textDocument/publishDiagnostics`; also use SourceKit's pull diagnostics (`textDocument/diagnostic`) for current unsaved Monaco content.
- When a feature is scoped down for version 1, also document the intended full version so MVP constraints do not become accidental product ceilings.
- Do not store app secrets in .NET user-secrets for this project. Use an external local JSON config file outside the project directory when file-based secrets are needed.
- Local AI review defaults should match the user's actual Ollama models, currently `gemma4:31b-mlx` and `qwen3.6:35b-mlx`; do not ship stale placeholder model names such as `llama3.1`.
- Form control rows must not inherit unrelated button spacing that breaks vertical alignment. Override global margins within dense control groups and verify select/button heights line up.
- The Node.js Servers curriculum track should target Fastify as the production framework after native Node boundaries are understood. Do not introduce Express.js into roadmap, content, or future task plans unless the user explicitly changes direction.
- For the Python/FastAPI curriculum, prefer larger coherent progression batches over tiny one-slice increments when the concepts naturally belong together. Maintain depth and verification, but move at a faster curriculum-building cadence.
