## TASK 0011 — Static analysis feedback

| # | Criterion | Verdict | Evidence |
|---|---|---|---|
| 1 | Runs against generated C# workspaces | ✅ | `RunStaticAnalysisAsync` runs a real `dotnet build` (`ExerciseRunner.cs:20-26`) on the copied run workspace (`ExerciseWorkspaceService.cs:130,137-138`); generated csproj enables `EnableNETAnalyzers=true`, `AnalysisLevel=latest` (`:753-754`). | 2 | Normalized for UI | ✅ | Build output → `StaticAnalysisDiagnosticResponse` records (`ExerciseWorkspaceService.cs:704-721`), relative path (`:716`), de-duped (`:719`), returned on run response (`:204-212`). |
| 3 | Severity, location, message, rule id | ✅ caveat | Regex captures all six (`:19-21,712-718`); DTO carries them (`:932-938`). **Caveat:** diagnostics without `(line,col)` (project-level/NETSDK/MSB errors) don't match the location-required regex and are **silently dropped**. |
| 4 | Stored with attempts/milestone validations | ✅ caveat | Full per-diagnostic rows persisted to `StaticAnalysisDiagnostics` (`:188-201`; entity `StaticAnalysisDiagnosticRecord.cs:1-26`; table `ContentIndexingHostedService.cs:186-212`); counts on `ExerciseRunHistory` (`:174-175`). **Caveat:** the detailed rows are **write-only** — no endpoint reads them back; only the live response + history counts reach UI/AI. |
| 5 | UI displays feedback separately from test failures | ✅ | Dedicated `StaticAnalysisPanel` (`ExerciseDetail.tsx:278-300`) separate from runner output (`:212`); history surfaces counts separately (`:315-316`); FE types match (`api.ts:197-203`). |
| 6 | Participates in project validation evidence | ⚠️ | Only error/warning **counts** flow into milestone AI review as advisory prompt text (`AiReviewService.cs:125-141,250-251`); prompt forbids a pass/fail gate (`:233`), policy `Advisory` (`:164`). No detailed diagnostics, no real validation gate. Ticket concedes this (lines 32-34). |

**Bugs / risks**
- **Fragile build-output regex.** `DotnetDiagnosticPattern` requires `(line,column)` (`ExerciseWorkspaceService.cs:19-21`) — location-less compiler/analyzer/SDK errors are lost; a build that fails with only such errors shows the learner **nothing** despite failing.
- **Localization fragility.** Regex hardcodes English `error|warning` (`:20`); no forced `DOTNET_CLI_UI_LANGUAGE`/UICulture, so parsing is host-locale dependent.
- **Performance: full `dotnet build` on every `/run`.** Build + visible test + hidden test are three separate `dotnet` invocations (`:138-142`), each 20s timeout (`ExerciseRunner.cs:18`); `dotnet test` rebuilds, so the project compiles ~twice per submission. No incremental/cached build; run workspace freshly copied + deleted each time.
- **Static analysis does not gate tests** — runs before but result ignored for sequencing; `RunVisibleTestsAsync` always runs even on build errors (`:140`). "Before tests" is positional only.
- **Persisted detail is dead data** — `StaticAnalysisDiagnostics` rows are never queried back.

**Test coverage:** Strong real E2E for the core claim (`ExerciseEndpointTests.cs:82-133`) — injects an undefined symbol, asserts a real `error` diagnostic + `StaticAnalysisErrorCount > 0` + summary in stored history; would catch a stub. Gaps: no **warning**-path test, no location-less/analyzer(CAxxxx) test (the drop risk is untested), no test of persisted rows, no `StaticAnalysisPanel` frontend test, no test that AI review receives counts.

**Verdict:** ✅ "Completed for exercise validation" largely defensible — analysis genuinely runs and is surfaced/persisted with test backing. Honest weak spots: counts-only advisory participation (crit. 6) and the brittle regex that silently drops location-less/localized diagnostics.

**Security findings**
- **Secret leakage — none found (PASS).** Secret loaded in-memory only, used solely as a `Bearer` header (`AiReviewService.cs:204-206`); response DTO exposes `SecretName` (the env-var name) never the value (`:353-360`); DB stores name only; no secret logging (grep clean); FE shows `secretName` only (`MilestoneDetail.tsx:131-132`).
- **SSRF — Low.** Server POSTs to `provider.Endpoint + /chat/completions` with no allowlist (`:202-203`), but there is **no API endpoint to create/modify a provider's `Endpoint`** — providers are seed-only (`api.openai.com`, `127.0.0.1:11434`), changeable only via direct DB access. Practical surface ~zero today; no guardrail if a provider-edit endpoint is added later.
- **No HTTP timeout / TLS hardening — Low.** `AddHttpClient<AiReviewService>()` (`Program.cs:16`) uses default 100s timeout, no explicit `Timeout`/retry/circuit policy; a slow/hostile endpoint can hold a thread up to 100s.

**Bugs / risks**
- **Empty-rubric silent pass** — missing `MarkdownPath` → `""` rubric (`ContentFileReader.cs:12-14`), review still records `RubricVersion="wordfreq-milestone-v1"`, falsely implying a rubric was used.
- **`RubricVersion`/`PromptVersion` are decoupled constants** (`AiReviewService.cs:16-17`) — don't change when the markdown/prompt changes; stored version can misrepresent what ran.
- **Shallow capability check** — `body.Contains("OK")` (`:62`) matches "OK" anywhere in the JSON envelope; doesn't verify `response_format: json_object` support (used at `:219`), so a provider can pass the test then fail at review time.

**Test coverage:** Good for an externally-dependent feature — 3 passing tests (`AiReviewTests.cs`): presets returned with no `sk-` leak (`:15-33`), test fails closed on missing secret (`:35-73`), full review against a mocked `FakeOpenAiHandler` storing structured advisory output with correct versions (`:75-148`). External calls properly mocked. Gaps: no SSRF/timeout test, no test that a *successful* provider test enables review, no empty-rubric test, no `Policy="Required"` test (crit. 10 unimplemented).

**Verdict:** ✅ "Completed for advisory milestone review" defensible. 8/10 fully met with passing tests; crit. 7 loose (rubric == milestone markdown) and crit. 10 structural-only — both scoped as future by the ticket. No secret leakage; SSRF unreachable via current API.

## Cross-cutting observations

- **App_Data pollution (carryover from 0008).** `src/ProdigeeTutsPoint.Api/App_Data/exercise-workspaces/` holds dozens of leftover test workspaces (`hover-test-*`, `format-test-*`, `runner-test-*`, …) — confirms the prior r2 finding that run/workspace cleanup does not fully reclaim disk. Not in scope for 0010–0012 but visible while reviewing them.
- **Recurring "scope-vs-claim" pattern.** Across all three tasks the engine/path is real and tested, but the ticket checkboxes read as fully general while the demonstrated coverage is one exercise (0010), one rubric (0012), or one diagnostic path (0011). The status *prose* is honest ("for the current C# exercise slice", "for exercise validation", "for advisory milestone review"); the `[x]` marks are not qualified.
- **No frontend tests for any of the new UI** (assistance panel, static-analysis panel, provider/review UI). Backend coverage is genuinely good; frontend coverage for these three features is zero.

## Recommended follow-ups (priority order)

1. **0011:** make build-output parsing robust — capture location-less diagnostics, force invariant culture (`DOTNET_CLI_UI_LANGUAGE`), and add warning-path + analyzer-rule tests. (Engineering risk: learners can get a failed build with no visible feedback.)
2. **0011:** either consume the persisted `StaticAnalysisDiagnostics` rows (history detail view) or stop writing dead data.
3. **0012:** add an explicit HTTP timeout and an endpoint allowlist/guard before any provider-edit endpoint ships; warn (don't silently proceed) on empty rubric; derive `RubricVersion` from rubric content.
4. **0010:** add tests for the mastery-evidence path (crit. 6) and the "solution after passing" branch (crit. 5).
5. **All three:** add frontend tests for the new panels; qualify the ticket `[x]` marks to match the honest status prose.
